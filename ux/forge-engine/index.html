Forge Engine

Forge Engine is a game engine created to develop a Year 2 and Year 3 game project.

It is also the game engine that came after Beta Engine, so I could take the opportunity to improve on what features I had implemented back then.

Howl: A Tale of Wolves, a DigiPen award-winning game, was created with Forge Engine.

Here are some of the tools I developed for Forge Engine:

C# Scripting API & Integration

Just like in Unity

One feature that Forge Engine has that stands out over Beta Engine is C# scripting support. It follows Unity's C# scripting API for the most part when it comes to function names. keywords and behaviors (Start, Update, OnDestroy). 

Many features seen in Unity C# scripting are supported, including OnCollision/Triggers, co-routines, and getting of other components/scripts.

Why

When developing for Spellthief in Beta Engine, there were overhead when it came to developing gameplay and UI behaviors, as every single change in code required a recompile via Visual Studios.

This overhead was not a big deal as all members working on Spellthief were technical people; we could recompile the project any time without any supervision. However, Forge Engine is also going to be used by designers. Getting them to recompile the project in Visual Studios can be a very time-consuming task.

Hence came the decision to implement a scripting API that the designers would be most familiar with: Unity's C# scripting. 

Prefab Interface

Here, we see apply and unlink in action

Forge Engine's prefab system supports most of what Unity's prefabs would support: applying, reverting, and unlinking. Linking is just a fun little function that links a transform to a prefab, essentially replacing it with an instance of the prefab.

Why

Beta Engine had a prefab tiler, which was sufficient for Spellthief as tilesets were used to build its levels. However, without using tilesets, the tiling functionality was useless. Instead, a need to save an object and all of its children as a prefab, duplicating it all over scenes, and having it retroactively change to replicate its master copy were required.

As Unity, a successful commercial game engine, had also implemented a prefab system, I decided to recreate most of their prefab system's features in Forge Engine.

File I/O API

Before, in Beta Engine

After, in Forge Engine

In Beta Engine, there was no file I/O API. As seen in the left image above, users who wanted to serialize their components had to literally type out tabs, new lines, opening and closing braces, etc. With provided feedback from users, I developed a robust file I/O API that can easily serialize and deserialize data.

No file paths, just a system name, property name and a value

In addition, I developed a Config Manager that can easily get and set properties without the users having to deal with file paths. They just have to state whether is it per project, user, project x user, or app data.

Why

When building APIs, the goal is to make life easy for the users. This file I/O API saves a lot of time and thinking when trying to read and write data to files.
